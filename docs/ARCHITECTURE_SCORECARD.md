## 优先级优化建议（按投入产出比）

### P0（建议本周）
1. **增加 step 重试策略（仅网络敏感步骤）**
   - 例如 020 邮件下载失败自动重试 2~3 次，指数退避。
2. **加“环境自检报告”**
   - `python main.py --check --report-file ...` 输出更详细的环境结果，方便远程排障。
3. **固定图片导图参数**
   - 把 `MAIL_IMAGE_COL_RANGE`、DPI、字体作为配置项，减少不同机器差异。

### P1（建议本月）
1. **把关键脚本函数化**（先 050 image / 050 mailtxt / 051）
   - 由 `main.py` 直接调用 Python 函数而非子进程，便于断点调试与单测。
2. **增加轻量测试**
   - `tests/test_validators.py`：输入/输出 pattern 校验。
   - `tests/test_steps_contract.py`：确保 steps 中的脚本文件都存在。
3. **配置分层**
   - 将硬编码常量迁移至 `config/*.yaml`（列范围、着色阈值、导图策略）。

### P2（中期）
1. **流程 profile 化**
   - `--profile=prod/dev/debug` 控制步骤开关（如是否生成图片、是否发邮件）。
2. **运行指标看板化**
   - 产出每步耗时、失败原因分布，便于长期优化。
3. **引入产物 manifest**
   - 每步写出 metadata（输入、输出、版本、耗时），形成可追溯链路。

## 给新手维护者的一句话

当前架构已经“能稳定跑 + 能定位问题”，下一阶段重点不是“大改重写”，而是做**可测试、可观测、可配置**的持续优化。

## 是否仍然高耦合？（结论）

**结论：从“高耦合”降到了“中等耦合”。**

原因：
- 以前：步骤依赖主要靠脚本内部约定，主调度层看不到完整依赖，属于“隐式耦合”；
- 现在：通过 `input_patterns` / `output_patterns` + `main.py` 前后校验，耦合关系已被显式化，属于“可管理耦合”。

仍未彻底低耦合的点：
1. 步骤间仍主要通过文件传递（Excel/HTML/PNG），不是内存对象契约；
2. 多个业务脚本对表结构列名有硬依赖（改表头会联动多步）；
3. 子进程式编排导致跨步骤调试与单测成本偏高。

建议目标：
- 近期保持当前模式（稳定优先）；
- 中期将 050/051 等关键步骤函数化并引入 manifest，进一步降耦合。

## “关键步骤函数化 + manifest” 是否有必要？

**结论：有必要，但建议“分阶段推进”，不是一次性重写。**

### 为什么有必要
1. **调试效率**：当前子进程串联方式在 PyCharm 里跨步骤断点不友好；函数化后可直接断点进函数。
2. **测试能力**：函数化后可以做真正单元测试（输入 DataFrame/Workbook，断言输出），回归成本下降。
3. **可追溯性**：manifest 让每步输入/输出、耗时、版本、关键参数可追踪，排障更快。

### 为什么不能“一步到位”
1. 这类 Excel 业务脚本对历史格式强依赖，重写风险高；
2. 当前流程已在生产跑通，稳定性优先；
3. 一次性函数化全部步骤，容易引入隐性行为差异。

### 推荐落地策略
- **第1阶段（现在）**：保留子进程模型 + 补 manifest（低风险）。
- **第2阶段**：只函数化 050 image / 050 mailtxt / 051（边界清晰，收益高）。
- **第3阶段**：再考虑前置数据加工步骤函数化（030/041/042）。

一句话：对这种 Excel 项目，函数化+manifest 是“值得做的中期工程”，但要“渐进式改造”。
